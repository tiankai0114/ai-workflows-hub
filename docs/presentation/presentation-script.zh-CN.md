# ai-workflows-hub 演讲口播稿

> 总时长建议：8～10 分钟正文 + 2 分钟 Q&A
> 受众：各 Team Leader、VP

---

## 第 1 页 · 封面（30 秒）

好，我们开始。

我今天要介绍的这个东西，用一句话概括就是：

**你写一个 Issue，加一个标签，Claude 自动帮你规划、写代码、做 Code Review —— 工程师只需要最后看一眼，点 Merge。**

这不是一个概念，是我们已经在用的东西，今天花 10 分钟给大家介绍一下它是怎么工作的，以及你们的 team 怎么快速接进来。

---

## 第 2 页 · 痛点（1 分钟）

先说我们为什么要做这个。

大家应该都有感受，工程师的时间很宝贵，但日常有很大一部分时间花在了重复性的工作上 —— 写 CRUD、修 bug、改一个小需求。这些事情不是不重要，但它消耗的是工程师最高价值的那部分精力。

第二个问题是 Code Review。PR 积压、排队等 Review、来回改，这个阻塞是很多 team 交付慢的核心原因之一。

第三个，大需求怎么拆，里程碑怎么定，这件事如果没有标准流程，每个人做法不一样，估时不准，计划混乱。

最后还有一个大家可能更关心的问题 —— AI 工具的安全合规。用第三方 AI 工具，代码有没有外传？API Key 有没有泄露风险？这些都是实际存在的顾虑。

这四个问题，我们这套方案全部解决了。

---

## 第 3 页 · 核心流程（1.5 分钟）

来看它是怎么工作的。

第一步，工程师或者 PM 用我们的标准模板创建一个 Issue，描述你想做什么。

如果需求很大，可以加一个 decompose 标签，Claude 会自动帮你拆成多个带依赖关系的子 Issue，不需要人工拆解。

然后加一个 `ready` 标签，触发 Claude Planner。它会自动扫描你的代码库，如果有不清楚的地方，会在 Issue 下面自动提问，最多三轮问答澄清完需求，然后输出一个结构化的里程碑计划。

**这里是第一个人工介入点** —— 工程师或 PM 确认这个计划方向没问题，加一个 `implement` 标签，授权 Claude 去执行。

然后 Claude Generator 开始按里程碑写代码，以 Bot 身份提交 commit、开 PR。PR 一开，Claude Evaluator 自动跑 build、lint、test，做代码审查，如果不通过，自动触发修复，最多修五轮。

通过之后，**第二个人工介入点** —— 工程师做最后的 Review，Merge PR。

整个过程，人只动了两次手。

---

## 第 4 页 · 人机协作（45 秒）

我想在这里特别强调一下这个"只介入两次"。

第一次是确认方向 —— Claude 规划好了，你看一眼，方向对就放行，方向不对就纠正，这是人最有价值的地方，做判断。

第二次是最终把关 —— AI 写的代码，质量已经经过了多轮自动校验，工程师做的是最后一道安全网，不是从头到尾的苦力活。

其余所有的工作 —— 需求拆解、代码扫描、问答澄清、代码实现、自动修复，全部由 AI 完成。

---

## 第 5 页 · 六大能力模块（1 分钟）

这套系统目前有六个模块。

前面讲的那个完整流程用到了 Decomposer、Planner、Generator、Evaluator 四个核心模块。

另外还有两个独立的能力：

一个是 **Code Review** —— 不只是 AI 写的 PR，工程师自己提的 PR，系统也会自动在上面留结构化的 Review 意见，帮 reviewer 提效。

另一个是 **CloudWatch Monitor** —— 定时拉取你们的 Lambda 或者服务日志，发现 ERROR 之后，Claude 自动分析根因，直接创建 Jira Issue 或者发 Teams 通知，不需要人盯着日志。这个对 SRE 和运维 team 很有价值。

---

## 第 6 页 · 技术架构（1 分钟）

简单说一下底层架构，这块对关心安全合规的同学比较重要。

AI 推理用的是 **AWS Bedrock**，模型是 Claude。所有的代码和上下文都在我们自己的 AWS 账号内处理，不经过任何第三方。

认证方式用的是 **GitHub OIDC** —— GitHub Actions 运行时动态获取一个临时凭证去 assume IAM Role，没有任何长期的 API Key 或者 Secret 存在代码或者 CI 环境里，从根本上消除了凭证泄露的风险。

接入方式是 GitHub 的 **Reusable Workflow** 机制 —— 你的 repo 只需要一行 `uses:` 引用我们这个 hub，不需要部署任何额外的服务，也没有额外的运维成本。

---

## 第 7 页 · 接入成本（45 秒）

那接入起来麻烦吗？我们评估过，整个过程 **20 分钟以内**。

就五步：复制几个模板文件，在 AWS IAM 里加一条白名单，创建一个 GitHub App 作为 Bot 账号，加三个 Secret，最后加五个 Workflow 文件引用 hub。

每个 Workflow 文件的核心配置就三个参数：你的 AWS Role ARN、Bot 的名字、Bot 的 ID。

20 分钟，一个工程师就能让整个 repo 具备 AI 辅助开发的能力。

这是我们已经接好的一个 repo。我现在创建一个 Issue，用标准模板填入需求描述。

**（创建 Issue，加上 `pge/status:ready` 标签）**

加上 ready 标签，大家看这里，GitHub Actions 已经自动触发了 —— Claude Planner 开始扫描代码库。

**（展示 Actions 运行界面）**

它在读代码、理解项目结构，如果有问题会回来问我们。计划出来之后，会作为 comment 贴在 Issue 下面。

**（展示生成的里程碑计划）**

这就是它输出的里程碑计划。我们确认方向没问题，加上 implement 标签。

**（加 implement 标签，展示 Generator Actions 触发）**

Generator 开始写代码了 —— 你们可以看到 Bot 在提 commit，PR 自动开出来，Evaluator 在跑 review。

这就是完整的流程，工程师全程没有动一行代码。
