项目特定功能的适配方案

novo-quote-ai 中的部分逻辑是 TypeScript / React 项目特有的，不适合写进通用 hub。 下面两个机制让每个消费者 repo 可以无侵入地注入自己的项目特定逻辑，同时 hub 保持对任意技术栈的适配能力。

💬
project_context 输入参数
注入项目特定的 AI 上下文，影响 Planner / Generator / Evaluator 的决策

每个 workflow 都暴露了一个可选的 project_context 字符串参数，会被插入到 Claude prompt 的最开头。消费者可以在这里描述：

  • 技术栈和架构约束（TypeScript + React、Kotlin Android、Python FastAPI 等）
  • 项目特定的评估维度（相当于 novo-quote-ai 的 G.6）
  • 模块边界规则和命名约定
  • 特殊的安全、性能或合规要求

novo-quote-ai 接入示例：

# pge-evaluate.yml (消费者 repo)
jobs:
  evaluate:
    uses: tiankai0114/ai-workflows-hub/.github/workflows/pge-evaluate.yml@v1
    with:
      project_context: |
        Tech stack: TypeScript, React 18, NestJS, PostgreSQL.
        G.6 layer checks (apply in addition to G.1–G.5):
        - React hooks must follow rules-of-hooks; no hooks in callbacks.
        - NestJS services must be injected via constructor, not new().
        - All DB queries must go through the Repository layer, not direct EntityManager.
🔧
pge-verify.sh 验证脚本
Consumer repo 提供真实 build/lint/test 验证，替代硬编码 CI 命令

在消费者 repo 创建 .github/scripts/pge-verify.sh，Evaluator 会在 Claude review 之前自动执行它：

  • 脚本退出 0 → Dimension A ✅，Claude 正常 review
  • 脚本退出非 0 → CI gate 阻断，自动打 pge/pr:needs-rework
  • 脚本不存在 → 降级为 Claude 读 PR 描述判断（向后兼容）

novo-quote-ai 接入示例：

#!/bin/bash
set -euo pipefail

# CHANGED_PATHS 由 hub 自动注入
while IFS= read -r path; do
  case "$path" in
    frontend/*) PROJECTS+=(frontend) ;;
    backend/*)  PROJECTS+=(backend)  ;;
  esac
done <<< "$CHANGED_PATHS"

for p in "${PROJECTS[@]}"; do
  pushd "$p"
  npm run typecheck && npm run lint && npm run test:unit
  popd
done
