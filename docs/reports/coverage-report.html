Adaptation Guide for Project-Specific Features

Some logic in novo-quote-ai is specific to TypeScript / React projects and does not belong in a generic hub. The two mechanisms below allow each consumer repo to inject project-specific logic non-invasively, while the hub remains compatible with any technology stack.

💬
project_context Input Parameter
Inject project-specific AI context, influencing Planner / Generator / Evaluator decisions

Every workflow exposes an optional project_context string parameter that is prepended to Claude's prompt. Consumers use it to describe:

  • Tech stack and architecture constraints (TypeScript + React, Kotlin Android, Python FastAPI, etc.)
  • Project-specific evaluation dimensions (equivalent to novo-quote-ai's G.6)
  • Module boundary rules and naming conventions
  • Special security, performance, or compliance requirements

Example — novo-quote-ai onboarding:

# pge-evaluate.yml (consumer repo)
jobs:
  evaluate:
    uses: tiankai0114/ai-workflows-hub/.github/workflows/pge-evaluate.yml@v1
    with:
      project_context: |
        Tech stack: TypeScript, React 18, NestJS, PostgreSQL.
        G.6 layer checks (apply in addition to G.1–G.5):
        - React hooks must follow rules-of-hooks; no hooks in callbacks.
        - NestJS services must be injected via constructor, not new().
        - All DB queries must go through the Repository layer, not direct EntityManager.
🔧
pge-verify.sh Verification Script
Consumer repo provides real build/lint/test verification, replacing hardcoded CI commands

Place .github/scripts/pge-verify.sh in the consumer repo — the Evaluator automatically executes it before Claude's review:

  • Script exits 0 → Dimension A ✅, Claude proceeds with review
  • Script exits non-0 → CI gate blocks, auto-labels pge/pr:needs-rework
  • Script absent → falls back to Claude reading the PR description (backward-compatible)

Example — novo-quote-ai integration:

#!/bin/bash
set -euo pipefail

# CHANGED_PATHS is injected automatically by the hub
while IFS= read -r path; do
  case "$path" in
    frontend/*) PROJECTS+=(frontend) ;;
    backend/*)  PROJECTS+=(backend)  ;;
  esac
done <<< "$CHANGED_PATHS"

for p in "${PROJECTS[@]}"; do
  pushd "$p"
  npm run typecheck && npm run lint && npm run test:unit
  popd
done
