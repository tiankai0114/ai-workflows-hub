🤖 ai-workflows-hub 接入指南

将 AI 驱动的开发工作流（PGE）集成到任意 GitHub Repo

7 个步骤，一次配置，后续 Repo 复用。完成后给 Issue 加标签，AI 自动写代码、开 PR、评审。

AWS Bedrock · GitHub Actions · Claude
⏱️
预计接入时间 15–20 分钟
✅
前提 AWS Role 和 GitHub App 已就绪
🔁
可复用 每个新 Repo 独立接入，互不影响

尚未配置 AWS Role 或 GitHub App？首次搭建额外需要约 10 分钟。

📁步骤 1

复制模板文件

将 library 中的标签、Issue 模板、CLAUDE.md 和 SKILL 文件复制到目标 repo。

  • 克隆 tiankai0114/ai-workflows-hub 到本地临时目录
  • 复制 templates/labels.yml → .github/labels.yml
  • 复制 templates/ISSUE_TEMPLATE/ → .github/ISSUE_TEMPLATE/
  • 复制 templates/CLAUDE.md.template → CLAUDE.md，填写项目描述
  • 复制 templates/cursor-skills/ → .cursor/skills/
# 克隆 library
gh repo clone tiankai0114/ai-workflows-hub /tmp/hub

# 进入目标 repo
cd /path/to/your-repo

cp /tmp/hub/templates/labels.yml .github/labels.yml
cp -r /tmp/hub/templates/ISSUE_TEMPLATE .github/ISSUE_TEMPLATE
cp /tmp/hub/templates/CLAUDE.md.template CLAUDE.md
mkdir -p .cursor/skills
cp -r /tmp/hub/templates/cursor-skills/clean-code .cursor/skills/
cp -r /tmp/hub/templates/cursor-skills/refactor .cursor/skills/
☁️步骤 2

配置 AWS IAM Role Trust Policy

在 AWS Console 找到用于 Bedrock 的 IAM Role，编辑 Trust relationships，允许目标 repo 通过 GitHub OIDC assume 该 Role。

  • 打开 AWS Console → IAM → Roles → 找到 Bedrock Role
  • 编辑 Trust relationships，加入以下 Statement
  • 把 YOUR_ACCOUNT_ID 和 YOUR_ORG/YOUR_REPO 替换为实际值
  • 确认 IAM Role 附加了 bedrock:InvokeModel 权限
{
  "Effect": "Allow",
  "Principal": {
    "Federated": "arn:aws:iam::YOUR_ACCOUNT_ID:oidc-provider/token.actions.githubusercontent.com"
  },
  "Action": "sts:AssumeRoleWithWebIdentity",
  "Condition": {
    "StringEquals": {
      "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
    },
    "StringLike": {
      "token.actions.githubusercontent.com:sub": "repo:YOUR_ORG/YOUR_REPO:*"
    }
  }
}
⚠️ sub 条件不能写 repo:org/*:*，必须精确到 repo 名称（AWS 要求至少 6 个字符前缀才能用通配符）。每新增一个 repo 都需要在此加一条记录。
🤖步骤 3

创建 GitHub App（Bot）

Generator 需要以 bot 身份提交代码和开 PR。Code Review、Decompose 和 Plan 不需要此步骤。

  • 打开 github.com/settings/apps/new
  • App name：填写全局唯一名称，建议加 repo 名前缀（如 your-repo-ci）
  • Webhook：取消勾选 Active
  • Permissions → Contents: Read & Write、Issues: Read & Write、Pull requests: Read & Write
  • Where installed：Only on this account
  • 点击 Create GitHub App，记录页面顶部的 App ID
  • 点 Generate a private key，下载 .pem 文件
  • 左侧菜单点 Install App → 安装到目标 repo
  • 查询 bot user ID（见下方命令），记录返回的 "id" 值
# 查询 bot user ID（替换 your-app-name）
curl https://api.github.com/users/your-app-name%5Bbot%5D | grep '"id"'

# 或者浏览器访问：
https://api.github.com/users/your-app-name[bot]
ℹ️ App name 中的空格要换成 -，全小写拼接 [bot] 后缀即是 bot username。
🔑步骤 4

在目标 Repo 添加 Secrets

打开目标 repo → Settings → Secrets and variables → Actions → New repository secret

Secret 名值必须
GH_APP_ID GitHub App 的数字 ID 必须 Implement / Evaluate
GH_APP_PRIVATE_KEY .pem 文件完整内容（含 -----BEGIN RSA PRIVATE KEY----- 头尾行） 必须 Implement / Evaluate
FIGMA_TOKEN Figma Personal Access Token 可选 有 Figma 设计稿时使用
ℹ️ Code Review、Decompose 和 Plan 只需要 AWS Role ARN（写在 workflow with 里），不需要 GH_APP_ID / GH_APP_PRIVATE_KEY。
🏷️步骤 5

导入 PGE 标签

将 PGE 标签体系导入到目标 repo，后续所有 workflow 触发都依赖这些标签。

  • 确认 .github/labels.yml 已在 repo 中（步骤 1 已复制）
  • 运行以下命令导入
# 确保 gh CLI 已登录
gh auth login

# 导入标签
gh label import .github/labels.yml --repo YOUR_ORG/YOUR_REPO
⚠️ 标签导入是 API 操作，不需要 commit 代码。但建议把 labels.yml 保留在 repo 里，方便后续重新导入或新成员查阅。
📝步骤 6

提交模板文件到默认分支

Issue Templates 必须在默认分支（main / master）上才会在 GitHub 的"New Issue"页面生效。

  • 将步骤 1 复制的文件全部 commit
  • 推送到默认分支（main 或 master）
  • 打开 repo → Issues → New issue，确认模板列表出现
git add .github/ISSUE_TEMPLATE .github/labels.yml CLAUDE.md .cursor/
git commit -m "chore: add PGE issue templates, CLAUDE.md and skill files"
git push origin main
⚙️步骤 7

添加触发 Workflow 文件

在 .github/workflows/ 下创建以下文件。替换三个占位符后 commit 到默认分支即可生效。

占位符替换为
YOUR_ROLE_ARNIAM Role ARN，如 arn:aws:iam::123456:role/my-bedrock-role
YOUR_BOT_NAMEGitHub App bot username，如 my-app-ci[bot]
YOUR_BOT_ID步骤 3 查到的数字 user ID
  • pge-code-review.yml 无需 GitHub App — 人工 PR 提交时触发代码审查
  • pge-decompose.yml 无需 GitHub App — Issue 加 pge/status:decompose 标签时拆分
  • pge-plan.yml 无需 GitHub App — Issue 加 pge/status:ready 标签时分析出计划
  • pge-implement.yml 需要 GitHub App — Issue 加 pge/status:implement 时写代码开 PR；PR 加 pge/pr:needs-rework 时自动修复
  • pge-evaluate.yml 需要 GitHub App — bot PR 打开/更新时评审；通过后推进下一里程碑
  • 将以上文件 commit 并 push 到默认分支
git add .github/workflows/pge-*.yml
git commit -m "feat: add PGE workflow triggers"
git push origin main
⚠️ pge-implement.yml 中的 evaluator_workflow_name 默认值为 "Claude: Evaluator"，与 pge-evaluate.yml 的 name: 字段保持一致即可。若你自定义了 evaluator 的 workflow 名称，需同步修改此参数，否则 Rework 模式下将无法触发 Evaluator 重新评审。

✅ 接入完成后的完整工作流

创建 Issue（使用 Issue Template）
↓ 加标签 pge/status:decompose
   Claude Decomposer 自动拆分为带依赖关系的子 Issue
↓ 给子 Issue 加标签 pge/status:ready
   Claude Planner 扫描代码库 → 自动问答（≤3 轮）→ 输出里程碑计划
↓ 确认计划 → 加标签 pge/status:implement
   Claude Generator 写代码 → 以 bot 身份开 PR
↓ PR 自动触发 pge/pr:approved 或 pge/pr:needs-rework
   Claude Evaluator build + lint + test + 代码审查 → 通过则推进下一 Milestone
↓ 人工 review 并合并 PR
